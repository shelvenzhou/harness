# Prompt Writing Guidelines

This directory is the source of truth for harness prompt text. Keep prompt
files small, role-aware, and explicit about when they apply. The runtime may
compose them with generated tool specs, model metadata, memory, and compacted
conversation summaries, so each file should own one layer of behavior.

## Composition Model

- `main.md` is the stable root orchestrator prompt. It defines identity,
  durable capabilities, response style, and hard safety rules.
- `playbook-<name>.md` is durable policy for a specific workflow. It should be
  injected with instruction-level authority, not treated as conversational
  memory.
- `role-<name>.md` is a subagent role suffix. It should describe that role's
  mission, allowed scope, output contract, and stopping conditions.
- Dynamic facts such as cwd, model/provider, sandbox, available tools, memory,
  and compacted summaries should be generated by code whenever possible.
- Tool schema descriptions are the primary source for exact argument contracts.
  Prompt files may explain when to use a tool, but should not duplicate every
  field unless that behavior is policy.

## Quality Bar

A good harness prompt is:

- **Scoped.** It says exactly which agent or workflow it governs.
- **Operational.** It tells the model what to do next, what evidence to gather,
  and when to stop.
- **Auditable.** It uses concrete acceptance criteria and output formats where
  the parent process or operator must make a decision.
- **Layered.** Stable identity, workflow policy, role behavior, and dynamic
  runtime facts live in separate sections or files.
- **Cache-friendly.** Static text belongs in the stable prefix; frequently
  changing memory and summaries belong in cache-tagged tail items.
- **Non-contradictory.** Higher-specificity playbooks override generic main
  guidance; role prompts should not fight the playbook that selected them.

## File Patterns

### `main.md`

Use `main.md` for durable behavior shared by the root orchestrator and normal
subagents:

- agent identity and what the harness runtime provides
- concise response style and progress-update expectations
- hard global safety rules
- generic tool-selection heuristics
- precedence rules for generated context and playbooks

Avoid:

- detailed workflows that only apply to one task family
- exact test command lists that belong to an implementer or playbook
- generated runtime facts such as current cwd, writable roots, or model name

### `playbook-*.md`

Use playbooks for repeatable workflows such as self-update, release, incident
response, or spawn strategy.

Each playbook should contain:

- **Trigger.** When the playbook applies.
- **Non-negotiables.** Safety constraints that must stop the agent if unmet.
- **Workflow.** A default sequence, with permission to adapt when warranted.
- **Verification.** Observable evidence required before claiming success.
- **Failure handling.** What to report, retry, or ask when blocked.
- **Out of scope.** Work the agent should not silently take on.

### `role-*.md`

Use roles to shape subagent behavior. A role prompt should be small enough that
the parent can predict the child output.

Each role should define:

- responsibility and ownership boundary
- allowed edits or explicit no-edit constraints
- required checks before final reply
- final reply format
- what counts as blocked or partial

Role prompts should assume the child may run either on the harness provider or
through a coding-agent CLI. Refer to capabilities by outcome first; use only
tool names that the child actually sees.

## Writing Rules

- Prefer direct imperatives over personality prose.
- Put the most important constraints before optional process advice.
- Use "must" only for rules that should be treated as policy.
- Include negative rules only when they prevent a real failure mode.
- Prefer concrete commands, paths, and event names over abstract wording.
- Keep examples short and representative; stale examples are worse than none.
- Do not encode secrets, private tokens, or environment-specific credentials.
- Do not make the model infer authority from headings. State precedence and
  conflict behavior explicitly.

## Review Checklist

Before merging prompt changes:

- The prompt belongs in the right layer: main, playbook, role, or generated
  runtime context.
- It does not duplicate a tool schema unless it adds decision policy.
- It names the trigger, evidence, and stopping condition for workflow text.
- It avoids conflicting with more specific playbooks or role contracts.
- It stays stable enough for prefix caching, or it is intentionally injected as
  dynamic tail context.
- There is a unit test or prompt dump check for any assembly behavior change.
